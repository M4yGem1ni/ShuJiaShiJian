# 浙江省线上扶贫募捐平台前后端设计与演示计划

## 一、项目定位

本项目是一个面向课程小组作业的线上扶贫募捐平台原型系统，目标不是投入真实运营，而是通过相对完整的前后端实现展示平台设计思路、核心业务流程、信任机制与资金透明化方案。系统应具备可运行、可演示、易讲解、开发成本低的特点，重点服务于答辩展示和项目文档支撑。

平台围绕浙江省数字公益与共同富裕主题，模拟政府、公益组织、捐赠用户、受助对象和平台管理员等多方角色，展示从扶贫项目发布、用户浏览捐赠、资金流向追踪、项目进展反馈到后台审核管理的完整闭环。

## 二、总体目标

### 1. 展示目标

- 展示一个“可看、可点、可讲”的线上扶贫募捐平台原型。
- 展示前台用户端与后台管理端的基本交互流程。
- 展示平台如何解决信任不足、参与门槛高、资金流向不透明等问题。
- 展示项目发布、便捷捐赠、资金追溯、效果反馈四个核心模块。
- 支撑课程报告中“功能架构设计、信任机制设计、技术实现与安全保障”等章节内容。

### 2. 开发目标

- 使用轻量、稳定、学习成本低的技术栈。
- 尽量减少复杂部署和真实第三方服务依赖。
- 使用模拟数据和本地数据库完成演示。
- 结构清晰，便于小组分工。
- 最终能够在本地一键启动或低成本部署到线上演示平台。

## 三、推荐技术栈

### 1. 前端技术栈

推荐使用：轻量级前端框架。

选择理由：

- Vue 3 学习成本较低，适合小组作业快速开发。
- Vite 启动速度快，项目结构简单。
- TypeScript 有利于减少数据字段错误，使代码更规范。
- Element Plus 提供表格、表单、弹窗、进度条、标签、菜单等后台和前台常用组件。
- ECharts 适合展示捐赠金额、资金流向、项目完成率等可视化数据，增强演示效果。

前端主要职责：

- 展示扶贫项目列表、项目详情、捐赠入口和反馈信息。
- 展示用户捐赠记录和资金流向时间线。
- 展示后台管理页面，包括项目审核、捐赠统计、反馈管理。
- 通过图表和时间线增强“透明公益”的视觉表达。

### 2. 后端技术栈

推荐使用：**Node.js + Express + TypeScript + Prisma + SQLite**

选择理由：

- Node.js 与前端同属 JavaScript/TypeScript 生态，方便小组成员理解。
- Express 简单灵活，适合快速搭建 REST API。
- Prisma 提供清晰的数据模型定义，数据库操作比手写 SQL 更直观。
- SQLite 不需要额外安装数据库服务，适合课程演示和本地运行。
- 后续如果需要部署，也可以迁移到 PostgreSQL 或 MySQL。

后端主要职责：

- 提供项目、捐赠、用户、资金记录、反馈等 API。
- 管理模拟登录与角色权限。
- 保存演示数据。
- 为前端提供统计数据和图表数据。

### 3. 数据库选择

推荐使用：**SQLite**

选择理由：

- 无需独立数据库服务器。
- 项目启动简单，适合课堂演示。
- 数据文件可随项目一起提交。
- 结合 Prisma 可以清楚展示数据表结构。

主要数据表：

- `User`：用户表，包含普通捐赠者、公益组织、管理员等角色。
- `Project`：扶贫项目表，包含项目标题、地区、目标金额、已筹金额、状态等。
- `Donation`：捐赠记录表，记录用户、项目、金额、时间和支付状态。
- `FundFlow`：资金流向表，记录资金从募集、拨付到使用的各阶段信息。
- `Feedback`：项目反馈表，记录项目进展、图片、文字说明和阶段成果。
- `AuditLog`：审核日志表，记录项目审核、资金审核和反馈审核过程。

### 4. 接口风格

推荐使用：**RESTful API**

原因：

- 简单直观，适合课程项目讲解。
- 与前端页面对应关系清楚。
- 便于展示“前后端”的技术架构。

示例接口：

- `GET /api/projects`：获取项目列表。
- `GET /api/projects/:id`：获取项目详情。
- `POST /api/donations`：提交模拟捐赠。
- `GET /api/projects/:id/fund-flows`：获取项目资金流向。
- `GET /api/projects/:id/feedbacks`：获取项目反馈。
- `POST /api/admin/projects`：后台新增项目。
- `PATCH /api/admin/projects/:id/status`：后台审核项目状态。
- `GET /api/admin/statistics`：获取后台统计数据。

## 四、系统角色设计

### 1. 普通捐赠用户

普通用户是平台前台的主要使用者，功能包括：

- 浏览扶贫项目。
- 按地区、类别、筹款进度筛选项目。
- 查看项目详情和受助背景。
- 进行模拟捐赠。
- 查看捐赠记录。
- 查看资金流向和项目反馈。

### 2. 公益组织用户

公益组织负责发起和维护项目，功能包括：

- 提交扶贫项目申请。
- 上传项目介绍、预算说明和证明材料。
- 查看项目审核状态。
- 发布阶段性进展反馈。
- 查看项目捐赠统计。

### 3. 平台管理员

管理员用于展示后台治理能力，功能包括：

- 审核项目发布申请。
- 审核项目反馈内容。
- 查看平台整体数据统计。
- 查看资金流向记录。
- 管理异常项目或风险提示。

### 4. 游客

游客无需登录即可访问基础内容，功能包括：

- 浏览公开项目。
- 查看平台介绍。
- 查看部分公开透明数据。

## 五、核心功能模块设计

### 1. 首页展示模块

首页用于快速体现平台主题和演示价值，应包含：

- 平台名称与宣传语，例如“浙江数字公益扶贫平台”。
- 关键数据卡片，例如累计捐赠金额、帮扶项目数、参与人数、覆盖地区数。
- 推荐扶贫项目列表。
- 浙江省公益成果展示区。
- 资金透明与可信机制介绍。

演示重点：

- 让老师第一眼看到项目主题清晰。
- 用数据卡片和图表提升真实感。
- 用“透明、可信、低门槛”突出设计亮点。

### 2. 扶贫项目列表模块

项目列表展示所有可捐赠项目，字段包括：

- 项目名称。
- 所属地区，例如杭州、衢州、丽水、温州等。
- 项目类型，例如教育帮扶、医疗救助、产业扶持、困难家庭救助。
- 目标金额。
- 已筹金额。
- 完成进度。
- 项目状态，例如募集中、执行中、已完成。

页面功能：

- 按地区筛选。
- 按项目类型筛选。
- 按项目状态筛选。
- 搜索项目名称。
- 点击进入项目详情。

演示重点：

- 展示扶贫资源的分类和精准匹配。
- 展示用户参与门槛低，容易找到感兴趣项目。

### 3. 项目详情模块

项目详情是前台展示的核心页面，应包含：

- 项目基本信息。
- 受助对象或地区背景。
- 项目目标与预算说明。
- 筹款进度条。
- 捐赠按钮。
- 资金流向时间线。
- 项目反馈列表。
- 审核与认证标识。

信任机制展示：

- 显示“平台审核通过”标识。
- 显示“公益组织认证”标识。
- 显示预算明细。
- 显示资金流转节点。
- 显示项目进展反馈。

演示重点：

- 让用户知道钱捐给谁、用在哪、进展如何。
- 体现平台对信任不足问题的解决方案。

### 4. 模拟捐赠模块

由于项目不需要真实使用，捐赠功能采用模拟支付方式。

页面流程：

1. 用户点击“立即捐赠”。
2. 弹出捐赠表单。
3. 用户选择或输入金额。
4. 用户选择是否匿名。
5. 点击确认捐赠。
6. 系统生成捐赠记录，并更新项目已筹金额。
7. 页面显示捐赠成功提示。

可选捐赠金额：

- 10 元。
- 50 元。
- 100 元。
- 其他金额。

演示重点：

- 不接入真实支付，避免复杂性和风险。
- 通过模拟数据展示完整捐赠流程。
- 捐赠成功后项目进度变化，增强演示效果。

### 5. 资金追溯模块

资金追溯是本项目的核心亮点之一。通过时间线和图表展示资金流向。

资金节点示例：

- 用户捐赠成功。
- 平台募集账户入账。
- 项目达到阶段目标。
- 资金拨付给公益组织。
- 资金用于采购物资或发放补助。
- 公益组织上传使用凭证。
- 平台审核并公开记录。

页面表现：

- 时间线展示每一笔资金状态。
- 饼图展示资金用途比例。
- 表格展示资金明细。
- 标签展示审核状态。

演示重点：

- 解决资金流向不透明问题。
- 展示“每一笔捐赠可追踪”的平台理念。
- 即使是模拟数据，也要让逻辑闭环清楚。

### 6. 项目反馈模块

项目反馈用于展示扶贫项目的实际效果。

反馈内容包括：

- 阶段标题。
- 发布时间。
- 反馈文字。
- 使用金额。
- 成果说明。
- 图片占位或示例图片。
- 管理员审核状态。

示例反馈：

- “第一批助学物资已送达”。
- “困难家庭生活补助已完成发放”。
- “乡村产业培训活动完成”。

演示重点：

- 让捐赠不是一次性行为，而是有后续反馈。
- 体现平台可持续运营与用户信任维护。

### 7. 用户中心模块

用户中心展示个人参与记录。

功能包括：

- 查看个人捐赠总额。
- 查看参与项目数量。
- 查看捐赠记录列表。
- 查看公益证书或感谢卡。
- 查看关注的项目。

可选设计：

- 生成一张简单的“电子公益证书”。
- 展示用户公益等级，例如爱心用户、公益伙伴。

演示重点：

- 增加用户参与感。
- 对应报告中的激励机制设计。

### 8. 后台管理模块

后台管理用于展示平台治理能力。

后台页面包括：

- 数据看板。
- 项目管理。
- 项目审核。
- 捐赠记录管理。
- 资金流向管理。
- 反馈审核。
- 用户管理。

数据看板展示：

- 平台累计捐赠金额。
- 项目数量。
- 各地区项目分布。
- 各类型项目占比。
- 近 7 日捐赠趋势。

项目管理功能：

- 新增项目。
- 编辑项目。
- 上下架项目。
- 审核项目。

演示重点：

- 体现平台不是单纯展示页面，而是有治理和审核机制。
- 支撑“平台治理、风控合规、数据驱动运营”相关文档内容。

### 3. 演示建议页面顺序

答辩演示时推荐按以下顺序：

1. 打开首页，介绍平台目标与核心数据。
2. 进入项目列表，展示项目筛选和分类。

8. 进入用户中心，展示个人捐赠记录和激励机制。

10. 总结平台如何回应需求痛点。

## 八、数据模型初步设计

### 1. User 用户表

字段建议：

- `id`：用户 ID。
- `name`：用户名称。
- `phone`：手机号，可使用模拟号码。
- `role`：用户角色，包含 donor、organization、admin。
- `avatar`：头像。
- `createdAt`：创建时间。

### 2. Project 项目表

字段建议：

- `id`：项目 ID。
- `title`：项目标题。
- `summary`：项目简介。
- `description`：详细说明。
- `region`：所属地区。
- `category`：项目类型。
- `targetAmount`：目标金额。
- `raisedAmount`：已筹金额。
- `status`：项目状态。
- `organizationId`：发起组织 ID。
- `coverImage`：封面图。
- `startDate`：开始日期。
- `endDate`：结束日期。
- `createdAt`：创建时间。

### 3. Donation 捐赠表

字段建议：

- `id`：捐赠 ID。
- `userId`：用户 ID。
- `projectId`：项目 ID。
- `amount`：捐赠金额。
- `isAnonymous`：是否匿名。
- `message`：留言。
- `status`：支付状态，演示中默认为 success。
- `createdAt`：捐赠时间。

### 4. FundFlow 资金流向表

字段建议：

- `id`：资金流向 ID。
- `projectId`：项目 ID。
- `title`：节点标题。
- `amount`：涉及金额。
- `type`：节点类型，例如 received、allocated、used、verified。
- `description`：说明。
- `proofUrl`：凭证地址，可为空或使用模拟图片。
- `status`：审核状态。
- `createdAt`：记录时间。

### 5. Feedback 反馈表

字段建议：

- `id`：反馈 ID。
- `projectId`：项目 ID。
- `title`：反馈标题。
- `content`：反馈内容。
- `usedAmount`：使用金额。
- `images`：图片列表。
- `status`：审核状态。
- `createdAt`：发布时间。

## 九、API 设计计划

### 1. 项目相关接口

- `GET /api/projects`：项目列表，支持筛选和搜索。
- `GET /api/projects/:id`：项目详情。
- `POST /api/admin/projects`：新增项目。
- `PUT /api/admin/projects/:id`：编辑项目。
- `PATCH /api/admin/projects/:id/status`：修改项目状态。
- `DELETE /api/admin/projects/:id`：删除或下架项目。

### 2. 捐赠相关接口

- `POST /api/donations`：创建模拟捐赠。
- `GET /api/users/:id/donations`：获取用户捐赠记录。
- `GET /api/admin/donations`：后台查看捐赠记录。

### 3. 资金流向接口

- `GET /api/projects/:id/fund-flows`：获取项目资金流向。
- `POST /api/admin/projects/:id/fund-flows`：新增资金节点。
- `PATCH /api/admin/fund-flows/:id/status`：审核资金节点。

### 4. 反馈接口

- `GET /api/projects/:id/feedbacks`：获取项目反馈。
- `POST /api/admin/projects/:id/feedbacks`：新增项目反馈。
- `PATCH /api/admin/feedbacks/:id/status`：审核反馈。

### 5. 统计接口

- `GET /api/statistics/home`：首页统计数据。
- `GET /api/admin/statistics`：后台统计数据。
- `GET /api/admin/statistics/trends`：捐赠趋势数据。

## 十一、信任机制设计

平台的信任机制需要在页面和数据中明确体现。

### 1. 项目准入机制

- 项目必须由认证公益组织提交。
- 项目上线前需要管理员审核。
- 项目详情展示审核通过标识。

### 2. 资金透明机制

- 每个项目展示目标金额和已筹金额。
- 捐赠记录可以在后台查看。
- 资金流转节点以时间线形式公开。
- 资金使用节点附带说明和凭证占位。

### 3. 反馈闭环机制

- 公益组织定期发布项目进展。
- 平台管理员审核反馈内容。
- 用户可在项目详情页查看反馈。

### 4. 用户激励机制

- 用户中心展示累计捐赠金额。
- 生成公益证书或感谢卡。
- 设置公益等级或徽章。

### 5. 风险提示机制

- 后台展示待审核、异常、已完成等状态。
- 对资金未及时反馈的项目显示提醒。
- 演示中可使用模拟风险标签。

## 十二、安全与合规设计说明

虽然项目是课程演示系统，但报告中仍需体现安全意识。

建议在设计中说明：

- 用户密码在真实系统中应加密存储，演示系统可简化登录。
- 接口需要进行角色权限控制。
- 用户手机号等敏感信息需要脱敏展示。
- 捐赠支付在真实系统中应接入合规第三方支付平台。
- 平台应保存审核日志，保证操作可追溯。
- 图片和凭证上传需要进行格式校验。
- 后台管理接口应限制管理员访问。

演示实现中可以简化：

- 使用固定账号模拟登录。
- 不接入真实支付。
- 不上传真实身份证、银行卡等敏感信息。
- 使用本地模拟图片和模拟数据。

## 十八、风险与应对

### 1. 时间不足

应对方式：

- 先完成前台核心演示流程。
- 后台功能可以以表格和模拟操作为主。
- 优先做视觉效果明显的模块。

### 2. 小组成员技术水平不一致

应对方式：

- 前端使用 Element Plus 降低页面开发难度。
- 后端使用 Express 和 SQLite 简化部署。
- 文档成员负责数据整理、截图和演示稿。

### 3. 后端联调困难

应对方式：

- 前端先使用 mock 数据开发页面。
- 后端接口完成后再替换 API。
- API 字段提前统一。

### 4. 演示时网络或环境异常

应对方式：

- 优先支持本地运行。
- 准备截图或录屏备用。
- 演示数据提前初始化。
